# Context Snapshot: newsletter-platform-mvp

- task statement: 폐쇄망 사내 웹 플랫폼의 첫 모듈로 뉴스레터/문서형 서비스 MVP를 설계하고, 이후 announcement/information/schedule/admin 모듈로 확장 가능한 modular monolith 기반 구현 계획을 수립한다.
- desired outcome: 실행 가능한 개발 계획 문서, PRD, 테스트 명세를 작성하고 이후 TDD 기반 구현에 바로 착수할 수 있는 단계별 산출물 구조를 확정한다.
- known facts/evidence:
  - 저장소 루트에는 현재 README.md, docs/, Newsletter/output/, .omx/ 만 존재한다.
  - 실제 운영 원본은 Markdown이 아니라 `Newsletter/output` 아래 HTML/PDF 파일이다.
  - `Newsletter/output` 에는 2026-02-06 ~ 2026-03-26 범위의 HTML/PDF 파일 쌍과 일부 `_debug.html` 파일이 존재한다.
  - 샘플 `newsletter_20260206.html` 은 인라인 스타일 기반 HTML 문서이며 외부 스크립트는 보이지 않고 외부 링크(anchor href)는 포함한다.
  - 사용자 요구 기술 스택은 Next.js + TypeScript + Tailwind CSS, FastAPI + Uvicorn + Pydantic, SQLite + Alembic + ORM, Docker Compose, 로컬 파일시스템 storage abstraction 이다.
- constraints:
  - 폐쇄망/오프라인 환경, 외부 SaaS 의존 최소화.
  - 초기 MVP는 HTML/PDF 파일 import/rendering 이 핵심이며 Markdown 은 구조 선반영 대상이다.
  - Windows 호스트 경로와 Docker 컨테이너 경로 차이를 고려해야 한다.
  - 구조는 newsletter 전용으로 굳지 않아야 하지만 과도한 마이크로서비스도 금지된다.
  - 관리자 인증, 경로 검증, path traversal 방지, HTML/Markdown 렌더링 보안이 필요하다.
- unknowns/open questions:
  - 실제 HTML 내부 상대경로 자산이 얼마나 있는지 전수 확인은 아직 안 했다.
  - 관리자 인증 방식(단일 로컬 관리자 시드 계정 vs 다중 사용자 운영) 세부 정책은 추후 구현에서 확정 필요.
  - PDF 썸네일 자동 생성은 MVP 필수 아님.
