실제 `_database`(뉴스레터·민간항공·문서·NSA)는 공개 배포 ZIP과 절대 같은 경로로 나가면 안 되고, 별도 승인·서명·암호 경계를 통과한 내부 번들로만 전달돼야 한다. 새 crypto 런타임을 도입하지 않고 검증된 Windows PowerShell 5.1의 .NET SignedCms/EnvelopedCms만 사용해 이 경계를 강제한다.

정책·경계 판정 로직(approval schema exactness, TTL≤24h, 역할 분리, trust policy digest 고정, ACL 형태, target 경로 봉쇄, inventory 무결성, EKU/AES OID 정확성)은 순수 Python 모듈로 분리해 실제 인증서·레지스트리 없이도 단위 테스트 가능하게 했다. PowerShell 계층은 실제 crypto I/O만 담당하고, 추출한 evidence(검증된 서명/체인 boolean, EKU OID, validity, ACL 항목)를 Python `--rpc` CLI에 넘겨 정책 결정을 재구현하지 않는다. `ConvertFrom-Json` 단독을 경계 검증으로 쓰지 않고, raw approval bytes를 duplicate-key 거부 strict parser로만 읽는다.

이중 역할 서명은 선언된 role 라벨이 아니라 서로 다른 thumbprint로 강제해, 같은 물리 서명자가 두 역할을 주장해도 거부한다. normal 번들은 `{newsletter,civil_aircraft,document}` subset + `include_nsa=false`, NSA 번들은 `{nsa}` only + `include_nsa=true`로 고정하고 혼합을 금지한다. importer는 decrypt 후 policy/signature/chain/EKU/validity/inventory/path를 전부 사전 검증하고, Task 3 maintenance gate 아래 same-volume staging·durable journal·old-root backup/rollback로만 target을 교체한다.

pydantic strict=True를 container/date 모델에서 뺀 것은 JSON의 str→datetime·list→tuple 정상 coercion을 strict가 거부하기 때문이며, field-level pattern/min_length 제약은 유지했다. 실제 trust provisioning과 real bundle은 이번 범위 밖으로, production trust absent 상태의 fail-closed 테스트를 기준으로 삼았다.
