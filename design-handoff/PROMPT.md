# 추천 핸드오프 프롬프트

이 폴더 전체를 새 Claude 세션에 첨부하고, 아래 프롬프트 중 하나를 그대로 붙여 넣으세요.

- **A. 전체 재디자인** — 8 페이지 전부를 새로 그릴 때

## A. 전체 재디자인 프롬프트

```
당신은 AeroOne 의 UI/UX 를 새로 설계하는 시니어 프로덕트 디자이너입니다.

본 폴더 (`design-handoff/`) 의 5 개 문서를 다음 순서로 30 분 안에 읽어주세요.
  1. README.md
  2. 01-design-brief.md
  3. 02-user-journeys.md
  4. 03-screen-inventory.md
  5. 04-design-constraints.md
  그리고 ../docs/images/ 의 현재 스크린샷 3 장 (list.png, preview.png, admin.png)

읽은 뒤 다음을 한국어로 제출해주세요.

## 산출물
1. **시각 언어 정의** — 색 팔레트 (라이트 / 다크 양 모드), 타이포 스케일, 간격 시스템, 모서리 / 그림자 토큰. Tailwind config 에 그대로 옮길 수 있는 형태.
2. **8 페이지 목업** — `03-screen-inventory.md` §A 의 8 페이지 전부를 라이트 / 다크 두 모드로. 우선순위는 §C 참고. 형식은 ASCII 와이어프레임 / HTML / Markdown 표 어떤 것이든 가능.
3. **컴포넌트 명세** — `03-screen-inventory.md` §B 의 18 컴포넌트 중 카드 · 폼 · 헤더 · 네비게이션의 4 가지 핵심 컴포넌트만 상태별 (default / hover / focus / active / disabled) 명세.

## 절대 제약 (위반 시 시스템 안 돔)
- 외부 폰트 / 외부 CDN / 외부 이미지 참조 0 (폐쇄망 운영). `04-design-constraints.md` §1 참조.
- 라이트 / 다크 양 모드를 반드시 동시 제출. 한 모드만 그리면 다른 모드가 미정의로 출시됨.
- 한 이슈 = 하나의 카드. HTML / PDF / Markdown 3 자산을 별도 카드로 분리 표시 금지.
- 단일 시드 관리자. 멀티 유저 권한 UI 그리지 마세요.

## 합의 절차
- `04-design-constraints.md` §6 의 "절대 피해야 할 것" 체크리스트 7 항을 깨야 한다면 시안 제출 전에 한 줄 메모로 사유를 먼저 알려주세요.
- `04-design-constraints.md` §5 의 기존 디자인 결정 9 건을 번복하려면 사유를 명시해주세요 (예: 달력 기본 접힘 결정, 홈 hero 제거 결정).

## 톤
저장소의 톤을 유지해주세요 — 건조한 우아함, 신뢰감, 본문이 주인공이고 UI 가 보조. 마케팅 카피 / 큰 hero / 일러스트레이션 금지.

준비되면 먼저 "이해한 핵심 제약 5 가지" 를 한 줄씩 5 줄로 적어 확인을 받은 뒤, 시안 작업을 시작해주세요.
```

## 대상 페이지
- [여기에 페이지 명시 — 예: `/newsletters` 목록 + `/newsletters/[slug]` 미리보기]

## 산출물
1. 대상 페이지의 라이트 / 다크 양 모드 시안 (ASCII 와이어프레임 또는 HTML 목업 가능).

3. 기존 시각 언어 (slate 팔레트, 시스템 폰트) 와의 정합성 확인. 새 토큰을 추가하면 명시.

## 절대 제약
- 외부 폰트 / 외부 CDN 참조 0.
- 양 모드 동시 제출.
- `04-design-constraints.md` §5 의 기존 디자인 결정 9 건을 번복하려면 사유 명시.

## 출력 언어
한국어.

준비되면 "이 페이지에서 가장 먼저 풀고 싶은 UX 문제 3 가지" 를 한 줄씩 3 줄로 적어 시작해주세요.

## C. 시각 언어만 새로 정의 (화면은 그대로)

```
당신은 AeroOne 의 시각 언어 (디자인 토큰) 만 새로 정의하는 시니어 디자인 시스템 엔지니어입니다.
화면 구조 / 레이아웃 / 컴포넌트 배치는 그대로 유지하고, 색 · 타이포 · 간격 · 모서리 · 그림자만 새로 제안해주세요.

본 폴더 (`design-handoff/`) 의 다음 2 개 문서만 읽으면 충분합니다.
  1. 01-design-brief.md (시스템이 만들어야 할 정서 — §4)
  2. 04-design-constraints.md (현재 팔레트 — §3.1, 절대 제약 — §1)

## 산출물
Tailwind config 의 `theme.extend` 에 그대로 옮길 수 있는 JSON 또는 TypeScript 객체:
- `colors` — 라이트 / 다크 양 모드의 background / surface / border / text-primary / text-secondary / accent / accent-hover / danger / success
- `fontFamily` — sans / mono (시스템 폰트 또는 npm 패키지에 묶이는 로컬 폰트만)
- `fontSize` — xs / sm / base / lg / xl / 2xl / 3xl 의 스케일
- `spacing` — 기본 4px 단위 위에 추가 토큰이 필요하면 명시
- `borderRadius` / `boxShadow` — 양 모드별

## 절대 제약
- 외부 폰트 금지 (Google Fonts / Adobe Fonts 등 외부 CDN 참조 0).
- 라이트 / 다크 양 모드를 반드시 동시 제출.
- 현재 본문 가독성을 깨지 않을 것 — 본문 텍스트 / 배경의 명도 대비 WCAG AA 이상.

## 출력 언어
한국어 설명 + 코드 블록은 영어 키.

준비되면 "이 시스템이 만들어야 할 정서 4 단어" (`01-design-brief.md` §4) 를 그대로 인용해 시작해주세요.
```

## 프롬프트 사용 팁

1. **첨부 방식** — 새 Claude 세션을 열고 `design-handoff/` 폴더 전체를 드래그하거나, GitHub URL 을 붙여 넣고 위 프롬프트 중 하나를 추가하세요.
2. **확인 게이트** — 모든 프롬프트가 작업 시작 전에 "확인 한 줄" 을 요구합니다 (핵심 제약 5 가지 / UX 문제 3 가지 / 정서 4 단어). 이 단계에서 디자이너가 제약을 잘못 이해하고 있으면 작업 전에 잡힙니다.
3. **반복** — 시안이 절대 제약을 위반하면 그대로 보여주고 "`04-design-constraints.md` §1 위반" 한 줄로 돌려보내면 충분합니다.
