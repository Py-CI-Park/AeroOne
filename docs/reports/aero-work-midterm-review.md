## 4. 근본 원인

**"체계적 엔지니어링"은 했으나 "체계적 요구사항 커버리지"는 빠졌다.**
- 착수 시 **실제 gongmuwon 앱을 항목별로 조사하지 않고**, 초기 계획서([`aero-work-plan.md`](../dev_plan/aero-work-plan.md))의 6기능 요약 매핑만 근거로 삼았다.
- 그 요약은 각 기능을 한 줄로 압축했을 뿐 실물 깊이(오케스트레이션·양식 엔진·지식위키·캘린더/알림·우측 패널)를 담지 못했다.
- 결과: **각 메뉴를 "존재하게" 만드는 데 최적화(폭)** 되어, gongmuwon 을 gongmuwon 답게 만드는 **연결·깊이(오케스트레이션)** 를 통째로 놓쳤다.
- 검증도 폭에 맞춰졌다 — 각 CRUD 는 테스트했으나 "대화→기능" 흐름은 애초에 구현이 없어 검증 대상도 아니었다.

## 5. 시정 조치

1. **진실 원천 재정의**: 실물 사용설명서를 항목 체크리스트로 고정([파리티 계획](../dev_plan/aero-work-full-parity-plan.md) §2). 이후 개발은 이 체크리스트를 수용 기준으로 삼는다.
2. **아키텍처 재편**: 탭 독립 CRUD → **인텐트 오케스트레이터 중심**(대화가 지식/일정/문서를 호출). 기존 스캐폴딩(지식폴더·일정·HWPX·실행기록)은 오케스트레이터의 **호출 대상 서비스로 재활용**한다(폐기 아님).
3. **단계 재계획**: F1(오케스트레이터) → F2(근거인용·이어가기) → F3(문서 양식 엔진) → F4~F5(지식위키) → F6(캘린더/알림) → F7(셸 UX) → F8(프로필·통합 QA). major(2.0.0) 트랙.
4. **정직한 상태 재보고**: "7/7 기능 완료"는 "**7메뉴 골격 완료, 실무 깊이·오케스트레이션 미구현**"으로 정정한다.

## 6. 권고

- 스캐폴딩은 버리지 않는다 — F1~F8 은 이 위에 얹는다.
- **F1(인텐트 오케스트레이터)** 이 가장 큰 가치이자 정체성이므로 최우선.
- 신규 대용량 의존(HWP/PDF 추출 wheel, 종이 미리보기 WASM)은 **오프라인 ZIP 반입 승인**을 선결.
- HWPX 서식은 **한컴 설치 PC 실기 검증**을 게이트로 둔다(본 개발 환경엔 한컴 없음).
- 범위상 단일 릴리스 금지 — 각 F단계 phase 보고서 + 다중 릴리스로 진행(AGENTS §9.6).
