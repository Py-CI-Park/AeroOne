# Aero Work 고도화 개발 로드맵 (1.18.0 이후)

- 작성일: 2026-07-20 (1.18.0 정식 릴리스 기준)
- 근거: 아키텍트 읽기 전용 분석 + `docs/reports/aero-work-vs-gongmuwon-final-review.md`(gongmuwon 대비 갭) + 실측 코드 상태
- 원칙: **기능 수 확대보다, 단일 사용자 데스크톱(gongmuwon) → 다중 사용자 웹 서버(AeroOne) 배포 모델 차이의 보정**을 먼저. 폐쇄망 순도(로컬 Ollama, 외부 egress 0)·단순성(SQLite·단일 워커) 유지.

> **핵심 진단**: Aero Work 는 폭과 기본 깊이가 충분하다. 다만 gongmuwon 의 단일 PC UX 를 기능 단위로 재현하면서 **자원 소유권·서버 파일 경계·AI egress 정책·실데이터 성능 예산**이 후순위가 됐다. 정식 고도화 전에 P0 로 먼저 닫는다.
>
> **1.18.0 릴리스 영향 없음**: 아래 P0 항목은 1.18.0 을 막지 않는다. Aero Work 는 현재 **admin 전용 카드**이고 이 동작들은 신규 도입이 아니라 기존 상태다. 다만 **일반 사용자 확대 전 반드시** 닫아야 한다.

## 하지 말 것 (과잉·폐쇄망 순도 위반·가짜 기능)

- ❌ 외부 OpenAI·클라우드 OCR·SaaS 임베딩/리랭커 호출 — 폐쇄망 순도 직접 위반, 지식·첨부·초안 외부 유출
- ❌ 수요·벤치마크 없이 멀티모달 모델 교체 또는 수 GB 모델을 설치팩 동봉 — 검증된 gemma4:12b 품질·230MB 패키지 장점 훼손
- ❌ 측정 전 Qdrant/Elasticsearch/Celery/Kafka·마이크로서비스 분리 — SQLite 단일 워커 단순성 상실(AW-R05 예산 실패가 먼저)
- ❌ gongmuwon Tauri 데스크톱 셸·픽셀 UI·화면 배율 컨트롤 복제 — 웹 모듈은 브라우저 줌·디자인 시스템으로 대체
- ❌ 한컴 실기 검증 전 WASM 미리보기 엔진 — 문제 미측정 상태에서 번들·라이선스·메모리 비용
- ❌ 파서·FTS·LLM 오류를 빈 문자열/LIKE 폴백으로만 조용히 숨기는 범용 fallback 확대 — 색인 누락·성능 회귀가 성공처럼 보임
- ❌ 비멀티모달 상태에서 이미지 업로드 버튼만 추가 — 가짜 지원
- ❌ 구형 HWP 지원 주경로를 한컴 COM 자동화·서버 GUI 설치 의존 — 서비스 계정·데스크톱 세션에 묶여 폐쇄망 서버 운영·자동 검증 취약

## 권장 착수 순서

1. **QW-1(운영자 한컴 매트릭스)** + **QW-2(egress 차단)** — 릴리스 직후, 저비용으로 유일 선결·순도 회귀 방지.
2. **AW-R01(폴더 소유권)** + **AW-R03(일정 삭제 확인)** — 일반 사용자 확대 전 필수 보안·안전.
3. **QW-3/QW-4/QW-5(측정)** → 그 결과로 **AW-R05(성능)·AW-R07(.hwp)·AW-R13(OCR)** 투자 여부 결정.
