## 3. 목표 아키텍처 (오케스트레이션 중심 재편)

gongmuwon 의 정체성은 "대화 한 줄이 일정·문서·지식으로 이어지는 것"이다. 따라서 각 탭을 독립 CRUD 로 두지 말고 **인텐트 오케스트레이터**가 중심에 서야 한다.

- **백엔드** `app/modules/aero_work/orchestrator/`:
  - `intent_router`: 발화 → 인텐트 분류(규칙 우선 + LLM 보조). gongmuwon 발화표(§2.1) 재현.
  - `intent_handlers`: schedule/document/knowledge/help 실행 + 멀티 인텐트 분해.
  - AeroAI(스트리밍·provider) + 지식폴더(P2) + 일정(P4) + 문서(HWPX) 를 **호출 대상 서비스**로 연결.
- **문서 엔진** `document/`: 양식별(시행문/1p/풀버전/이메일/임의형식) 개조식 구조 생성기 + OWPML 양식 템플릿 + 승인 상태기계.
- **지식위키** `knowledge/`: FTS(SQLite FTS5 트라이그램) + 본문 추출기(HWP/HWPX/PDF/DOCX) + 위키 파생(분류체계·업무허브·버전 가족).
- **프런트**: 셸에 우측 컨텍스트 패널 + 온보딩 + 탭별 심화 뷰. AeroOne 디자인 토큰으로 gongmuwon UX 를 재현(픽셀 클론 아님).

## 4. 단계 계획 (F1~F8, 우선순위 순)

각 단계는 **AeroOne 자산 매핑 + 신규 + 검증 게이트 + phase 보고서**를 포함한다.

- **F1 — 인텐트 오케스트레이터 ✅ 구현됨**: 발화 → 인텐트 분류(규칙 기반) + 한국어 자연어 날짜·시각 파싱 + 멀티인텐트 분해 → 일정(등록/조회/삭제)·문서작성 인식·도움말·지식검색 라우팅. `korean_datetime`·`intent_router`·`orchestrator_service`(기존 Schedule/Knowledge 서비스 재사용) + `POST /orchestrate`(CSRF·실행기록) + `WorkChatPanel`(명령+결과 카드+예시 힌트, 업무대화 탭). 검증: 발화표(§2.1) 단위 13(파서 7+분류 6) + 통합 7(등록→조회→삭제·문서인식·도움말·멀티인텐트·지식 graceful). LLM 보조 라우팅은 후속.
- **F2 — 근거 LLM 합성 + 최신본 판별 ✅ 구현됨**: 지식 인텐트가 지식폴더(P2) 검색 결과를 (1) `version_ranker` 로 파일명 날짜·버전·'최종' 표식을 읽어 **최신본 판별**(같은 문서 다판본에서만 표시), (2) 로컬 LLM(`OllamaClient` 재사용)으로 **출처 표기가 붙은 근거 답변 합성**(best-effort, 실패 시 근거만). `OrchestratorService` 에 embedder/synthesizer 주입 → 테스트는 실 LLM 없이 결정적. `WorkChatPanel` 이 근거 답변 + 각 근거의 '최신본' 배지를 렌더. 대화→일정/문서 이어가기는 F1 에서 이미 실행(일정 등록·문서 인식→다운로드). 검증: 최신본 단위 5 + 오케스트레이터 지식 단위 2(주입). 스트리밍 합성은 후속.
- **F3 — 문서작성 엔진 ✅ 기본 구현**: **양식 5종**(1페이지 □요약/□섹션/◦항목 · 시행문 수신/제목/1.·가./끝 · 풀버전 Ⅰ.·Ⅱ. · 이메일 제목/인사/서명 · 임의형식) 개조식 구조 포매터(`document_formats`, 규칙 기반·결정적) + `hwpx_generator.build_hwpx_document`(문단 리스트 직접 조립) + 문서작성 API 에 `format` 슬롯 + `DocumentPanel` 양식 선택기 + 업무대화 문서 인텐트가 인식한 양식으로 다운로드. 검증: 양식 단위 6(위계 기호) + HWPX 구조 6 + 문서 통합 4(시행문 위계 포함). **LLM 개조식 구조 생성·미리보기 수정지시·승인 상태기계·종이(WASM) 미리보기는 후속.**
- **F4 — 지식폴더 추출·키워드 검색 ✅ 기본 구현**: 본문 추출을 **PDF(pypdf)·DOCX(python-docx)·HWPX(네이티브 zip+xml)**로 확장(`text_extract`, best-effort). **wheel 반입은 pypdf·python-docx 만**(HWPX 는 OWPML 이라 네이티브 처리로 wheel 최소화); `pip download` 가 lxml 등 공유 의존을 패키지당 단일 wheel 로 dedup → **오프라인 ZIP 중복 없음(실증: pypdf·python_docx·lxml·typing_extensions 각 1개)**. **키워드 검색**(`keyword_search`, LIKE 기반·Ollama 불필요) + `POST /knowledge/keyword-search` + `KnowledgePanel` 의미/키워드 토글. 파일 상한 20MB 상향. 검증: 추출 단위 5(DOCX·HWPX·PDF·손상 graceful) + 키워드 단위 1 + 통합 1. **HWP(구 바이너리)·FTS5 트라이그램·상세검색 스트리밍·색인 품질/구조 보기는 후속.**
- **F5 — 지식위키(버전 가족) ✅ 기본 구현**: 색인된 문서를 **버전 가족**(같은 문서의 대표=공식본 + 옛 판본 이력)으로 묶어 보여주는 gongmuwon 업무 허브 백본(§6.5). `version_ranker.group_by_family`(base 그룹핑 + 대표 판별, F2 재사용) + `KnowledgeService.wiki` + `GET /knowledge/wiki` + `KnowledgeWiki`(대표 앞세우고 판본 이력 접기/펴기)를 지식폴더 탭에 배치. 검증: 가족 그룹핑 단위 1 + 통합 1(2판 가족 대표=최신 + 단일 문서 미표시). **분류체계 마법사(LLM 업무 분류)·주제 페이지·문서 카드(요약/품질)·증분 이동 승계/소프트삭제는 후속.**
- **F6 — 일정 캘린더/알림 ✅ 기본 구현**: **월 캘린더 그리드**(6주, 이전·오늘·다음, 날짜별 일정 블록·🔔·오늘 강조) + 아젠다/월 토글 + **사전 알림**(이벤트 `remind_before_minutes` 컬럼 + 마이그레이션 `20260719_0023` + 없음/10/30/60/1440 선택) + **인앱 알림 배너**(`ReminderBanner`, 리드타임 진입 시 "곧 시작 N분 후", 30초 폴링·확인 시 숨김). `ScheduleMonth`·`ReminderBanner` 컴포넌트. 검증: 사전 알림 왕복 통합 1(생성 30→수정 60→해제 null) + 마이그레이션 up/down 가역(sim 실증). **주/일 시간축 뷰·토스트·우측 '가까운 일정' 배지는 F7(우측 패널)과 함께 후속.**
- **F7 — 전역 셸 UX ✅ 기본 구현**: **우측 컨텍스트 패널**(`AeroWorkContextPanel`, xl 이상)에 업무 엔진 상태(AI status dot)·가까운 일정(🔔)·지식 색인 요약·최근 실행기록을 얹고 항목 클릭 시 해당 탭 이동. **첫 실행 온보딩**(`AeroWorkOnboarding`, 3단계: 로컬 AI 연결·지식폴더 등록·첫 대화 — 상태에서 파생, AI ok+폴더 등록 시 자동 숨김·'나중에'로 dismiss). 셸을 온보딩 배너 + 3열(메뉴/본문/컨텍스트)로 재편. 프런트 전용(검증된 조회 API 재사용), tsc 0. **주/일 시간축 뷰·대기 승인 패널·헤더 배율/토스트는 후속(승인은 F 문서 승인 상태기계와 함께).**
- **F8 — 통합 QA + phase 보고서 ✅ 기본 구현**: aero_work 백엔드 **80 passed** + 프런트 tsc 0 + 마이그레이션 `20260719_0020~0023` 가역 + wheel dedup 실증을 통합 게이트로 확인하고, F1~F8 완료 보고서([`../reports/aero-work-phase-f1-f8.md`](../reports/aero-work-phase-f1-f8.md))에 산출·검증·후속·릴리스 스코프(major 2.0.0)를 정리. per-user LLM 프로필 전환(사용자 선호 테이블 설계 필요)·성능 예산·브라우저 E2E·한컴 실기 검증은 릴리스 선결 게이트로 후속 처리 대상이었고, G007에서 브라우저 E2E·전량 게이트·ZIP QA를 닫았다. 한컴 실기 검증은 운영자 게이트로 남는다.

| 주제 | 결정/조사 |

| **키워드 검색** | SQLite **FTS5**(트라이그램, 한국어) — 현 코사인(의미검색)과 병행 |
| **종이 미리보기** | @rhwp/core(WASM)는 gongmuwon 참조. AeroOne 은 우선 서버 렌더 근사(HTML) → WASM 은 2차 |
| **알림** | 폐쇄망이라 인앱 전용(배너·토스트·배지) + 브라우저 Notification(옵션). 이메일/푸시 없음 |
| **승인형 정책** | 문서 최종 저장 등 되돌리기 어려운 작업에 '대기 중인 승인' 상태 — AeroOne 감사 로그와 정합 |

## 7. 검증 전략

- 각 F단계: 백엔드 단위+통합 테스트, 프런트 tsc 0 + 컴포넌트 테스트, 실 Ollama 필요분 실증(F1 인텐트, F2 근거인용).
- 오케스트레이터: gongmuwon 발화표(§2.1)를 **회귀 케이스로 고정**(발화→인텐트→동작).
- 문서 양식: OWPML 구조 유효성 + 양식별 위계 기호 스냅샷. 한컴 실기 호환은 운영자 실기 게이트.
- 릴리스: 전체 게이트(backend pytest·frontend vitest·오프라인 ZIP) + phase 보고서.

## 8. 결론

현재는 **골격(breadth)** 만 섰다. 완전 파리티는 **F1 인텐트 오케스트레이터**(정체성)부터 채워야 하며, F2(근거인용·이어가기)·F3(문서 양식 엔진)·F4~F5(지식위키)가 실무 가치를 만든다. F6~F8(캘린더·셸 UX·프로필)이 마감이다. 신규 대용량 의존(HWP/PDF 추출·WASM)은 **오프라인 ZIP 반입 승인**을 선결하고, HWPX 서식은 한컴 실기 검증을 게이트로 둔다. 전체는 **major(2.0.0)** 트랙으로 다중 phase 진행한다.
